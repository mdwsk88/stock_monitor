# Stock Monitor 业务流程图

这份文档从“业务运行”和“用户触达”视角描述 `stock_monitor`，重点不是代码模块，而是：

- 数据如何进入系统
- 系统如何筛选、理解和组织事件
- 报告、实时预警、手动演示、MCP 问答分别怎么流转

如果你要对外介绍项目，可以把这份文档理解为“业务版架构图”。

## 1. 业务总览

项目当前主业务围绕 A 股展开，核心目标是把分散的公告和宏观快讯，转化成几类可以直接消费的结果：

- 盘前早报
- 盘后复盘
- 盘后风险速递
- 盘中实时预警
- 宏观实时推送
- 企业微信机器人问答
- 手动演示推送

## 2. 总体业务流程

```mermaid
flowchart LR
    Source["外部信息源\nA 股公告 / 宏观快讯 / 政策新闻"]
    Clean["业务理解层\n去噪 / 分类 / 多空判断 / 信号评分"]
    Aggregate["结果组织层\n事件聚类 / 个股聚合 / 主题映射 / 共振融合"]
    Decision["输出决策层\n状态机 / 冷却 / 协调 / 推送策略"]
    Report["日报输出\n盘前早报 / 盘后复盘 / 风险速递"]
    Realtime["实时输出\n盘中预警 / 宏观实时 / 市场脉冲"]
    MCP["研究输出\nMCP 个股 / 榜单 / 主线问答"]
    Demo["演示输出\n手动测试台 / Demo Push"]
    User["最终用户\n群成员 / 运营 / Agent"]

    Source --> Clean
    Clean --> Aggregate
    Aggregate --> Decision
    Decision --> Report
    Decision --> Realtime
    Aggregate --> MCP
    Decision --> Demo

    Report --> User
    Realtime --> User
    MCP --> User
    Demo --> User
```

## 3. 自动化日报业务流程

这条链路负责把过去一段时间内的重要事件整理成固定时点输出。

```mermaid
flowchart TB
    T1["定时任务触发\n盘前 / 盘后 / 盘后风险时点"]
    T2["读取时间窗口内的公告与宏观事件"]
    T3["筛选高价值事件\n过滤低信息量公告"]
    T4["个股聚合与分栏\n机会 / 风险 / 主线 / 共振"]
    T5["AI 生成日报内容\n中文或英文"]
    T6["企业微信 Markdown 推送"]
    T7["群成员阅读日报"]

    T1 --> T2 --> T3 --> T4 --> T5 --> T6 --> T7
```

### 这条流程产出的典型业务结果

- `A股盘前早报`
- `A股盘后复盘`
- `A股盘后风险速递`

## 4. 盘中实时预警业务流程

这条链路强调“高价值、低噪音”，不是所有公告都推，而是只对高分机会或高危风险事件即时吹哨。

```mermaid
flowchart TB
    R1["高频抓取 A 股公告"]
    R2["公告事件分类与评分"]
    R3["盘中有效性校验\n是否交易时段 / 是否高分 / 是否白名单事件"]
    R4["市场状态校验\n防守态 / 进攻态 / 高潮态"]
    R5["推送协调\n冷却时间 / 多空互斥 / 家族级静默"]
    R6["生成盘中预警卡片"]
    R7["企业微信群推送"]

    R1 --> R2 --> R3 --> R4 --> R5 --> R6 --> R7
```

### 这条流程的业务价值

- 不是把所有公告刷到群里
- 而是把“值得盘中盯一眼”的事件变成结构化卡片
- 让群成员能快速看到结论、推演、主线共振和风险提醒

## 5. 宏观实时推送业务流程

这条链路关注的是政策、流动性、产业和风险冲击，不依赖单只股票公告。

```mermaid
flowchart TB
    M1["宏观快讯定时回扫"]
    M2["宏观事件去噪与主题识别"]
    M3["判断利多 / 利空 / 重要级别"]
    M4["结合市场状态与推送协调层"]
    M5["生成宏观实时消息"]
    M6["企业微信群推送"]

    M1 --> M2 --> M3 --> M4 --> M5 --> M6
```

### 典型业务场景

- 政策发布
- 流动性宽松 / 收紧
- 监管边际变化
- 地缘或商品价格冲击

## 6. 手动演示业务流程

这条链路专门用于测试和对外演示，不依赖定时任务，也不强依赖真实盘中数据。

```mermaid
flowchart LR
    U1["运营 / 演示人员"]
    U2["手动测试台\nmanual-push-console"]
    U3["选择语言\n中文 / English"]
    U4["选择动作\n日报 / 晚报 / 风险 / Demo Intraday Push"]
    U5["手动接口\nReportPushController"]
    U6["真实推送链路\nWeComApi"]
    U7["企业微信群显示结果"]

    U1 --> U2 --> U3 --> U4 --> U5 --> U6 --> U7
```

### 这条流程有两个业务特点

- 可以立即验证功能，不必等待定时任务
- 可以用 mock 数据走真实企业微信发送路径，适合现场展示项目效果

## 7. 企业微信机器人问答业务流程

这条链路不是“推送”，而是“查询式使用”。

```mermaid
flowchart TB
    Q1["群成员 @ 机器人提问"]
    Q2["MCP Server 接收问题"]
    Q3["根据问题选择工具\n个股 / 风险榜 / 机会榜 / 主线 / 共振"]
    Q4["查询结构化事件结果"]
    Q5["按机器人 Prompt 组织回答"]
    Q6["返回群内研究结论"]

    Q1 --> Q2 --> Q3 --> Q4 --> Q5 --> Q6
```

### 适合回答的问题

- 某只股票最近偏多还是偏空
- 今天有哪些高分机会股
- 哪条宏观主线最强
- 哪些股票和主线有共振

## 8. 业务角色视图

从业务角色看，这个系统同时服务 3 类人。

```mermaid
flowchart LR
    Ops["运营 / 开发\n手动测试台\n看板 / 群晖部署"]
    Trader["群成员 / 研究员\n日报 / 盘中预警 / 风险提示"]
    AgentUser["机器人用户 / Agent\nMCP 问答 / 工具调用"]
    System["Stock Monitor"]

    Ops --> System
    Trader --> System
    AgentUser --> System
```

### 对不同角色的价值

- 对运营：方便测试、演示、排查和发版验证
- 对群成员：更快获得可执行的信息摘要和风险提醒
- 对 Agent：把研究能力标准化成可调用工具，而不是只能看群消息

## 9. 推荐讲解顺序

如果你要拿这份流程图给别人介绍项目，建议按这个顺序说：

1. 先讲“总体业务流程”，让对方知道系统在做什么
2. 再讲“自动化日报流程”和“盘中实时预警流程”，突出主价值
3. 接着讲“手动演示流程”，说明为什么这个项目便于展示
4. 最后讲“MCP 问答流程”，说明它不只是推送系统，还是一个研究能力平台
